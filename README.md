# **개인 개발 리포트 (CODI-IT — Store/Product API, Deploy)**

## 1. **프로젝트 개요**

**프로젝트명**: **CODI-IT**

**프로젝트 목적**:  
CODI-IT은 셀러와 바이어가 함께 사용하는 패션 이커머스 플랫폼입니다.  
본 프로젝트의 백엔드는 상품/스토어/주문/결제/배송/리뷰/문의/알림까지 이어지는 핵심 도메인을 안정적으로 제공하고, 운영 환경(EC2 기반)에서 실제 배포·장애 대응이 가능한 구조를 만드는 것을 목표로 했습니다.

**핵심 기능**:

- 스토어 등록/수정/조회, 관심 스토어 등록/해제
- 상품 CRUD, 카테고리/사이즈/재고/할인 관리
- 사이즈 가이드(의류/신발) 및 재고 연동
- 주문/결제/배송/리뷰/문의/알림 API
- Swagger 기반 API 문서 제공
- GitHub Actions + EC2 기반 자동/수동 배포 운영


## 2. **담당한 작업**

**역할**: Store / Product API + 배포 운영

**기여 내용**:

- Store API 파트

  - 스토어 등록/수정/조회 기능 구현
  - 관심 스토어 등록/해제 기능 구현
  - 판매자 약관/개인정보 동의 UI 연동(구매자 노출 항목 안내 및 동의 플로우 반영)
  - 스토어 사업자 정보(사업자등록번호, 통신판매업 신고번호 등) 유효성 검증
  - 스토어 민감정보 암복호화(AES-256-GCM) 적용 및 마스킹/보관 정책 반영

- Product API 파트

  - 상품 CRUD 전체 구현
  - 상품 공시 정보 필드 확장(Product/Store) 및 카테고리별 고시 항목 반영
  - 상품 재고(ProductStock), 사이즈 스펙(ProductSizeSpec) 설계 및 API 연동
  - 카테고리별 사이즈 가이드 분기(TOP/BOTTOM/SHOES/NONE) 적용
  - 사이즈 선택 정책(FREE 단독, 의류 XS~XL, 신발 mm 단위) 정비
  - 프론트-백엔드 사이즈 매핑 이슈 대응(동적 매핑)

- 배포/운영 파트

  - GitHub Actions 배포 파이프라인 운영
  - EC2 수동 배포(cherry-pick, branch sync, pm2 restart) 절차 정리
  - `npm ci` lockfile 불일치/환경 차이 이슈 복구
  - 디스크 Full(용량+inode) 장애 대응 및 로그/캐시 정리 자동화 방향 수립


## 3. **기술적 성과**

**사용 기술 스택**:

- Backend: Node.js / Express / TypeScript
- Database: PostgreSQL / Prisma ORM
- Validation: Superstruct
- Infra/Deploy: AWS EC2, RDS, S3, Route 53, Nginx, PM2, GitHub Actions
- 협업 도구: GitHub, Notion, Discord


**주요 성과**:

1) 스토어/상품 도메인 안정화

  - 사업자 정보 검증 로직을 서비스 레이어에 통합해 잘못된 데이터 입력 차단
  - 전자상거래 고시/신원정보 표시 의무에 맞춘 공시 정보 구조 확장
  - 상품 생성/수정 시 사이즈·재고·카테고리 검증을 강화해 데이터 정합성 확보
  - 스토어 민감정보 암호화 저장으로 보안 요구사항 반영

2) 사이즈 체계 확장 및 실사용 UX 개선

  - FREE, XS~XL, SHOES(230-290, 5단위) 체계를 API/시드/프론트까지 일관 적용
  - 사이즈 매핑을 고정 ID 의존에서 동적 매핑 구조로 보완하여 운영 DB 변화 대응
  - 상품 상세/주문/리뷰 흐름에서 사이즈 표기 일관성 개선

3) 운영 배포 대응력 강화

  - 브랜치 diverged, cherry-pick 충돌, `npm ci` 실패, 디스크 Full 장애를 복구 절차로 체계화
  - 수동 배포와 자동 배포 간 싱크 전략을 정리해 운영 리스크 축소

4) 인프라 구성요소 이해 및 운영 적용

  - AWS 기반으로 서비스 계층을 분리해 운영(애플리케이션/DB/정적 파일)
  - EC2(앱 실행), RDS(PostgreSQL), S3(이미지 저장), Route 53(DNS), Nginx(리버스 프록시), HTTPS(TLS) 구성을 일관되게 운영
  - SSH 기반 원격 배포/장애 복구 절차 확립
  - 구성요소별 책임을 명확히 분리해 장애 원인 파악과 복구 속도 개선


## 4. **문제점 및 해결 과정**

1) 스토어 악용 대응 체계(StoreAuditLog + 보관정책 스케줄러)

- 상황: 판매자가 스토어를 생성한 뒤 악의적 상품 판매 후 거짓 정보를 가진 스토어 변경/탈퇴하는 경우, 운영 측에서 사후 대응/역추적이 어려운 리스크 존재
- 원인: 변경 이력 스냅샷과 보관 정책 집행 체계가 없으면 추적 근거가 부족함
- 해결:
  - 전자상거래 고시/신원정보 표시 의무에 따라 판매자 노출 정보(상호, 대표자, 사업자번호, 통신판매업 신고번호, 사업장 주소, 연락처)와 상품 고시 항목(소재, 제조국, 제조연월, 품질보증 기준, A/S 책임자/연락처)을 구조적으로 반영
  - 스토어 생성/수정 시점의 스냅샷을 `StoreAuditLog`에 저장해 변경 이력 추적 가능하도록 구현
  - 감사 로그에는 민감정보가 직접 남지 않도록 마스킹 처리 적용
  - 탈퇴 회원 개인정보는 30일 보관 후 null 처리(소프트 삭제 정책 반영)
  - `StoreAuditLog`는 1년 보관 후 삭제 정책 반영
  - 위 정책 집행을 위해 스케줄러를 적용하고, 매주 월요일 새벽 3시에 실행되도록 운영 자동화
- 성과:
  - 스토어 변경 이력 기반 역추적 가능성 확보
  - 악용 시나리오 대응 근거 데이터 확보
  - 개인정보 보관/파기 정책을 코드 레벨에서 일관되게 집행

2) 사이즈 기능 확장 중 `ProductStock_sizeId_fkey` 오류 발생

- 상황: 상품 생성/수정 시 사이즈 ID가 운영 DB와 불일치하여 500/P2003 오류 발생
- 원인: 프론트가 고정된 사이즈 ID 매핑을 가정하고 전송
- 해결:
  - 서버에서 `sizeId/sizeName` 기반 정규화 로직 강화
  - 프론트에서 응답 데이터 기반 동적 매핑 적용
  - FREE 단독/의류 사이즈 묶음 정책 분기 적용

3) 배포 단계에서 `npm ci` 반복 실패

- 상황: EC2 배포 시 `package-lock.json` 불일치(`fsevents`)로 파이프라인 중단
- 원인: 실행 환경(Node/NPM)과 lockfile 해석 차이
- 해결:
  - 서버 환경에서 lockfile 재생성 및 `npm ci` 재검증
  - 로컬/서버 버전 동기화 후 커밋 반영

4) EC2 디스크 Full로 git fetch/pull 실패

- 상황: `No space left on device`, `unpack-objects failed`로 배포 실패
- 원인: npm 캐시, 빌드 캐시(.next), 로그 파일 누적으로 루트 볼륨(8GB) 포화
- 해결:
  - `~/.npm`, `.next/cache`, PM2/nginx 로그 정리
  - inode 사용량까지 함께 점검
  - 로그 로테이션/정기 정리 적용 계획 수립
 

## 5. **협업 및 피드백**

- PR 기반 리뷰 및 피드백 반영으로 API 계약/응답 형태를 지속 정리
- 배포 이슈 시 팀 단위로 브랜치/환경 상태를 빠르게 공유하고 복구 절차 통일
- 도메인 경계(Store/Product/Order) 충돌 지점을 조기에 확인하는 방식으로 협업 개선

배운 점:

- 기능 완성도만큼 운영 환경 일관성(Node/NPM/lockfile)이 중요함
- 프론트-백엔드 계약에서 “ID 고정 가정”은 운영에서 쉽게 깨질 수 있어 동적 매핑이 안전함
- 실서비스 관점에서는 장애 복구 절차(runbook)가 개발 속도만큼 중요함


## 6. **코드 품질 및 최적화**

단일 책임 원칙 준수:

- module/controller/service/repository/structs/utils 분리
- 검증 로직(Struct + service util)과 비즈니스 로직 계층화

일관된 응답/에러 처리:

- `BadRequest/Unauthorized/Forbidden/NotFound/Conflict` 공통 에러 체계 사용
- Superstruct 검증 실패 시 400 + `issues` 반환으로 디버깅 가능성 강화
- 프론트 토스트에서 백엔드 `message` 노출하도록 오류 전달 개선

성능/운영 최적화:

- 이미지 URL 해석/조회 시점 정규화
- 불필요한 재빌드/재배포 반복을 줄이기 위한 수동 배포 표준 순서 정리
- 스토어 개인정보 30일 파기 + 감사로그 1년 보관 정책을 스케줄러로 자동화


## 7. **향후 개선 사항 및 제안**

스토어/상품 도메인 고도화:

- 사이즈 정책 메타데이터 API(카테고리별 허용 사이즈/가이드) 표준화
- 에러 `issues`를 프론트 폼 필드에 직접 매핑하는 UX 개선
- 상품/스토어 수정 이력(audit) 조회 API 보강

배포/운영 고도화:

- Node/NPM 버전 고정(.nvmrc + CI setup-node)으로 lockfile 이슈 재발 방지
- PM2/nginx 로그 로테이션 + 캐시 정리 자동화
- EBS 증설 및 모니터링(디스크/메모리/프로세스) 알람 연동
